# AI-first dashboard — agent command center

**Date:** 2026-07-02
**Branch:** `feat/ai-first-dashboard` (off `origin/main`, isolated worktree)
**Status:** Approved by Amin (direction A of three options)

## Problem

The practice dashboard reads as an old-school CRM stats page: seven KPI cards plus
passive lists (unread, hot leads, appointments, campaigns, activity). Every widget
reports the past and asks the human to figure out what to do. Meanwhile the codebase
has a deep AI layer (setter/closer agents, agent loop with real confidence, autopilot
with escalations, predictive scoring, no-show risk) that is invisible on the front
page. The escalations API (`/api/autopilot/escalations`) has **zero UI consumers**.

## Direction

Invert the page's job: from "here are your numbers" to "here's what your AI did,
and what it needs from you." Activation, not net-new — v1 adds no new AI capability,
only surfaces decisions the system already makes.

## Layout (top to bottom)

1. **Header** — compact greeting + autopilot status pill
   (`organizations.autopilot_enabled/paused` + count of `messages` where
   `ai_generated = true`, outbound, today). Pill links to autopilot settings.
2. **Ask bar** — slim input; on submit it swaps in the existing `CommandCenter`
   component (extended with an `initialMessage` prop) so the first question is
   sent immediately. No new AI surface.
3. **Today's brief** — 3–4 sentence narrative card, serif (`aurea-display` /
   editorial style). Generated by `claude-haiku-4-5` from **code-derived facts
   only** (LLM writes prose, code decides actions). New module
   `src/lib/ai/daily-brief.ts`, budget-gated via `isUnderDailyBudget`, recorded
   via `recordAiUsage` with new feature `'daily_brief'` (no DB constraint on the
   column; TS union extended). In-memory 15-min cache per org. On any failure →
   deterministic code-composed fallback sentence; the page never breaks.
4. **Needs you (decision queue)** — left ⅔ column. Card sources, all existing data:
   - **Drafted replies**: `escalations` (status `pending`) with `ai_draft_response`
     + `ai_confidence`. Actions: *Approve and send* (POST `/api/sms/send`
     `{lead_id, message}` → PATCH escalations `resolve`), *Open lead* (edit path),
     *Dismiss* (PATCH `dismiss`). Approve only offered when a draft exists;
     escalations without drafts render as "needs review" with Open lead.
   - **No-show risks**: `appointments` next 72h, unconfirmed,
     `no_show_risk_score ≥ 60`. Actions: *Call* (`tel:`), *View appointment*.
   - **Going cold**: hot leads, no contact/response in 72h. Actions: *Call*, *Open lead*.
   - Empty state: "All clear — I'm watching N active conversations" (never blank).
5. **Agent activity** — right ⅓ rail. `lead_activities` filtered to a curated list
   of AI-initiated types (`ai_scored`, `ai_speed_to_lead`, `escalated_to_human`,
   `cross_channel_sms_sent`, `cross_channel_email_sent`, `financing_link_sent`),
   plus today's schedule mini-card. Live inserts via Supabase Realtime
   (`postgres_changes`, pattern from `lead-timeline.tsx`).
6. **Metrics strip** — the seven KPIs collapse to one row of quiet chips, each
   linking to its full page. Nothing lost, no longer the headline.

## Files

  risky appointments, stale hot leads, AI activity, autopilot status, AI sends today);
  compute facts; call brief generator server-side.
- `src/lib/ai/daily-brief.ts` — new; follows `summarize.ts` conventions.
- `src/components/crm/dashboard-home.tsx` — rewritten as command center; split into
  focused pieces (header/brief/queue/activity/strip) in
  `src/components/crm/dashboard/` if the file grows past ~350 lines.
- `src/components/crm/command-center.tsx` — add optional `initialMessage` prop only.
- `src/lib/ai/usage.ts` — add `'daily_brief'` to `AiUsageFeature`.

## Non-goals (v1)

- No new tables or migrations.
- No new autonomous behavior — every queue action goes through existing gated APIs
  (consent, quiet hours, kill switch all still apply).
- Ask bar answers come from the existing `command-agent.ts`; no prompt changes.
- Realtime for the queue itself (server-rendered; refresh on action) — only the
  activity feed subscribes live.

## Testing

- `tsc --noEmit` must pass (type errors block Vercel deploys on main).
- Production build green.
- Visual verification via local dev server: populated state and empty state
  (org with no escalations) both render; approve/dismiss round-trip works against
  dev data.

## Risks

- Aurea visual language must be preserved (stone/zinc/emerald, Instrument Serif,
  hairline borders) — the redesign changes information architecture, not the theme.
- `appointment_scheduled` / `stage_advanced` activity types are logged by both staff
  and autopilot paths; excluded from the AI feed in v1 to avoid mislabeling.
