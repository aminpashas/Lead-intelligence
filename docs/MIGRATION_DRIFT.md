# Migration Drift — Assessment & Remediation Plan

**Date:** 2026-06-27
**Prod project:** Lead Intelligence — `bnmnirzfwopqklsitjgq`
**Status:** Documented. **No destructive action taken** — reconciliation needs a human decision on approach (see options below).

## TL;DR

The repository's `supabase/migrations/` directory is **not** the source of the
production schema. They are two unrelated namespaces:

| | Repo (`supabase/migrations/`) | Prod (`supabase_migrations.schema_migrations`) |
|---|---|---|
| Count | **77 files** | **64 tracked versions** |
| Naming | hand-numbered: `001_…`, `038_…`, then `20260617_…` | timestamped: `20260408005745` … `20260627213747` |
| Origin | hand-authored in the repo | generated by dashboard / `db diff` / `apply_migration` |

Because the names don't correspond, **a fresh environment built from the repo
files cannot reproduce prod**. This was proven directly: a Supabase preview branch
(which replays the repo files) reached `MIGRATIONS_FAILED` after 17 files and was
missing objects prod has — e.g. `agency_admin` (repo migration 018), the
`is_admin_role()` function (repo migration 020), and the real `user_profiles_*`
policy names.

This is a **DR / staging / CI risk**, not a live security hole. Prod itself is fine.

---

## Evidence

- Prod tracked migrations (64): all timestamp-versioned, `20260408005745` →
  `20260627213747`. The last three were added cleanly by this audit via
  `apply_migration` (`20260627213747` = C1; the R1/R2 migration registered same-day).
- Repo files (77): `001_organizations_and_users.sql` … `20260627_user_profiles_*`.
- Branch replay of the repo files: **failed at 17**, missing 018+/020+ objects.
- Live prod RLS on `user_profiles` uses policy names (`user_profiles_update`,
  `user_profiles_delete`, …) that **do not appear in any repo migration file**
  (the files say `"Users can update their own profile"` / `"Admins can manage
  user profiles"`). Confirmed the schema was built/edited outside the repo files.

## Why it happened

Migrations were applied to prod ad-hoc — via the Supabase dashboard SQL editor,
`supabase db diff`-generated timestamped files, and/or MCP `apply_migration` —
rather than by pushing the hand-numbered repo files through `supabase db push`.
Over time the repo files became design notes while the DB evolved independently.
Likely compounded by migrations authored on feature branches that were applied to
prod before/without merging.

## Risks

- **No reproducible schema:** staging, preview branches, and disaster recovery
  cannot rebuild prod from version control.
- **Branch-based CI is unreliable** (replay fails), so schema changes can't be
  validated on an ephemeral copy — we had to fall back to isolated harness tests
  and prod pre-flight for the C1/R1/R2 fixes.
- **Onboarding/foot-guns:** a new dev reading the repo sees policies/functions
  (`is_admin_role`, admin-manage policy) that aren't actually on prod, and vice versa.

## Remediation options

### Option A — Re-baseline from the live DB *(recommended)*
Adopt production as the source of truth and squash to a single baseline.
1. `supabase db pull` (or `pg_dump --schema-only`) against `bnmnirzfwopqklsitjgq`
   to capture the real schema as one `…_baseline.sql`.
2. Move the 77 legacy hand-numbered files to `supabase/migrations/_archive/`
   (keep for history; they no longer replay).
3. Going forward, author every change as a timestamped migration applied via
   `apply_migration` (already proven working — C1/R1/R2 registered correctly).
4. Verify: create a throwaway branch → it should now replay the baseline cleanly.

*Pros:* fast, gives a reproducible schema immediately. *Cons:* loses granular
migration history (acceptable — it's already lost in practice).

### Option B — Reconcile file-by-file
Map each of the 77 repo files to a tracked prod version, `supabase migration
repair` the `schema_migrations` table to match, and patch the gaps. *Pros:*
preserves history. *Cons:* high effort, error-prone, and several repo files
clearly never matched what was applied (different policy names) — so it's partly
guesswork.

---

## Recommended next step

Do **Option A** in a branch and confirm a clean replay before changing the repo on
`main`. This is a deliberate, human-in-the-loop operation (it rewrites the
migrations directory), so it's intentionally left for explicit go-ahead rather
than automated here.

## 🚨 NEW HAZARD: a local file can be OLDER than the live object

The 2026-06-27 assessment framed drift as "prod has things the repo lacks" — a
*replay* problem, mostly dormant. This sweep found a sharper, active failure:

**A repo migration whose filename sorts LAST can still be an OLDER definition
than what is live.** Applying it "to catch prod up" silently regresses prod.

Concrete, hit on 2026-07-22: `reclassify_off_funnel_contacts` had three
`create or replace` migrations in prod and only the first had a local file.

```
20260722194758  reclassify_off_funnel_v2              local file (named 20260722120000_…)
20260722194930  …_status_guard                        NO local file
20260722200805  …_prior_visit_guard                   NO local file  <- LIVE
```

Applying the local base v2 would have stripped the live prior-visit guard
(`exists (select 1 from ehr_appointments …)`), which is what stops a prospect who
booked through LI from being parked out of the funnel. Captured as `3981fca`.

**RULE: never apply a migration to prod on filename evidence alone.** For any
`create or replace` / constraint / policy, diff the live definition first:

```sql
select pg_get_functiondef(oid) from pg_proc where proname = '<fn>';
select pg_get_constraintdef(oid) from pg_constraint where conname = '<name>';
```
